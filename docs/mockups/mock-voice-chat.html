設計メモ

実装時の技術的な参考情報

アーキテクチャ（案B強化版）

┌─ MyAiba 音声会話モード ──────────────────────────────┐ │ │ │ [マイク] → Web Speech API (STT) │ │ ↓ テキスト確定 │ │ [Lambda] → Bedrock ConverseStream (Haiku 4.5) │ │ ↓ テキスト差分（WebSocket chat_delta） │ │ [ElevenLabs] Streaming TTS → 音声チャンク │ │ ↓ │ │ [スピーカー] + Live2D リップシンク │ │ │ │ 記憶: AgentCore Memory ✅ / 永久記憶 ✅ / 要約 ✅ │ │ スキル: Calendar / Weather / Places / Tasks ✅ │ │ 想定レイテンシ: ~1.5秒（STT確定→音声再生開始） │ └───────────────────────────────────────────────────────┘

画面遷移フロー

  • /aiba マイAi-Baタブ → 「話しかける」ボタン
  • /aiba/voice 音声会話画面（フルスクリーン、ダークテーマ）
  • 会話終了 → サマリー画面（トピック保存 or マイAi-Baに戻る）
  • トピック保存時 → /themes/{themeId} へ遷移（テキストチャットで続行可能）

TTS: ElevenLabs ストリーミング

  • WebSocket API: テキストチャンクを逐次送信 → 音声チャンク（base64 MP3）を逐次受信
  • LLM の chat_delta を句点で区切って ElevenLabs に流す
  • 初回音声チャンクが届いた時点で即再生開始（バッファリング不要）
  • 音量エンベロープ: 既存の computeVolumeEnvelope をチャンク単位に拡張
  • 日本語対応ボイス: Multilingual v2 モデル（eleven_multilingual_v2）
  • APIキー管理: SSM Parameter Store → Lambda 環境変数

STT: Web Speech API（Phase 1）

  • 既存の useSpeechRecognition hook をそのまま活用
  • continuous: true + VAD でハンズフリー会話
  • 確定テキストを即座に Lambda に送信
  • 将来: Whisper API に切替可能（精度向上・iOS対応強化）

Live2D 連携

  • 音声会話中もキャラクターの表情・モーションは通常通り制御
  • LLM の JSON レスポンスから emotion / motion を抽出
  • ElevenLabs 音声再生中: 音量エンベロープで口パク（リップシンク）
  • 音声会話画面ではキャラクターを大きく表示（h-[60vh] 程度）

会話セッション管理

  • 音声会話はメインセッション扱い（sessionId を共有）
  • 会話中のメッセージは通常通り DynamoDB に保存（user/assistant 各 MSG#）
  • 5ターンごとのローリング要約も通常通り起動
  • 会話終了時: トピックとして保存可能 → テキストチャットで続行
  • サマリー画面: 会話要約 + 保存された記憶 + 感情の流れを表示

コスト見積もり

  • ElevenLabs: $0.15〜0.30/1000文字（Starter $5/mo = 30,000文字）
  • 1回の音声会話（5分, 8ターン）: AI側 ~800文字 → ~$0.12〜0.24
  • 1日2回 × 30日 = 月~48,000文字 → Scale プラン $22/mo（100,000文字）推奨
  • Bedrock (Haiku 4.5): 既存コストと同等（ストリーミングでも料金変わらず）

実装フェーズ

  • Phase 1: 基本の音声会話ループ（STT → Bedrock Stream → ElevenLabs Stream → 再生）
  • Phase 2: 会話終了サマリー + トピック保存 + Live2D連携
  • Phase 3: PC版レイアウト（トランスクリプトパネル）
  • Phase 4: 音声カスタマイズ（キャラクターごとの音声設定）
